# Diff Review

Pre-ship AI review (Phase 4). The sanitizer IS the feature: sensitive paths
excluded by name, every line through redaction (see [[security-enforcement]]),
budgets with visible truncation, injection suspects flagged independently of
the model, per-run random fence with the untrusted-data rule.

Dogfood story worth remembering: on its first real run against this repo, the
feature **found a bug in its own freshly-written UI** — a blocked-only change
set (e.g. a lone `.env` edit) never reaches the model, yet the panel showed the
green "ship it". Fix: ship-it requires `filesReviewed > 0`. Moral: dogfood
before declaring a gate passed.

Operational notes: sonnet sometimes answers in prose → parser degrades to a
visible raw block (never silent); output contract is restated at the prompt
tail; CLI timeout 360s. The per-block review action in the terminal reuses the
same boundary via `review.runText` — blocks come from the app-level store
(Phase 3.1), the seam [[swarm-design]] reviewer roles will reuse.
- (2026-07-04) AI review now has two entry surfaces sharing one renderer (src/components/ReviewFindings.tsx): the GitPanel 'Review before ship' card (review.run) and a per-block bridge in BlocksView (review.runText with `$ cmd\noutput`), both routed through the same sanitizer boundary. Non-obvious coupling: the UI marks a finding as a prompt-injection suspect purely by checking if its title contains 'prompt-injection' (isInjectionFinding), so a model-authored finding can be misattributed as sanitizer-detected. parseFindings was hardened to accept markdown fences, a {"findings":[…]} object root, uppercase severities, and numeric-string lines.
- (2026-07-04) The pre-ship AI review (VISION 4.5/4.6) surfaces in two places, both rendered by the shared src/components/ReviewFindings.tsx: (1) GitPanel 'Review before ship' card via cockpit().review.run(projectId) over the working tree, and (2) a per-block bridge in BlocksView via cockpit().review.runText(projectId, {label, content}) that reviews a captured command+output. The block→review path runs through the SAME sanitizer boundary as a diff review (secrets masked, prompt-injection suspects flagged). Convention: findings whose title contains 'prompt-injection' get the warm warning treatment in the renderer (revfind--injection) — the renderer keys off the title string, so the sanitizer must emit that phrasing.

- (2026-07-04) Renderer panels keyed on activeProjectId reset their state via a useEffect when the project changes. Any in-flight async IPC call that closes over activeProjectId (e.g. GitPanel.runReview → cockpit().review.run, or the BlocksView block→review bridge → review.runText) will, if it resolves AFTER the user switches projects, silently repopulate the NEW project's panel with the OLD project's findings/error — corrupting a trust surface like the pre-ship review gate. Fix: follow the existing `cancelled`/in-flight flag pattern already used in the canRefreshApp effect so a late response for a now-stale project is dropped. Surfaced by the AI pre-ship reviewer while building the VISION 4.5/4.6 review feature.
